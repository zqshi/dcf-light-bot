# OpenClaw 人机协同设计文档

> 版本：v1.0 | 日期：2026-04-01 | 状态：草案

## 1. 背景与问题

### 1.1 当前状态

OpenClaw 当前是"工具模式"——人下指令，AI 执行，人等结果。这本质上是把 AI 当搜索引擎用。

### 1.2 核心问题

| 维度 | 工具模式的问题 |
|------|--------------|
| 人的角色 | 操作者，逐条下指令，效率受限于操作速度 |
| AI 的角色 | 被动执行器，没有上下文，每次交互独立 |
| 信息流 | 指令→结果，线性单向，没有决策环节 |
| 效率瓶颈 | 人的操作速度，而非人的决策质量 |

### 1.3 目标

升级为"管理者模式"——人做决策，AI 自主执行。用户的价值在于**判断和决策**，而不是逐条下指令。

| 维度 | 管理者模式 |
|------|-----------|
| 人的角色 | 决策者，定义目标和边界 |
| AI 的角色 | 自主执行者，主动汇报，生成推荐方案 |
| 信息流 | 目标→进展→决策点→结果，异步闭环 |
| 效率瓶颈 | 人的决策质量（AI 负责效率） |

## 2. 协同模型：三层授权

人和数字员工的协同，核心是**授权边界**。不是所有事都需要 AI 自主，也不是所有事都需要人审批。

```
┌─────────────────────────────────────────┐
│  Layer 3: 自主执行（人不介入）           │
│  例：日常数据汇总、日志分析、周报生成     │
│  条件：低风险、可逆、有模板              │
├─────────────────────────────────────────┤
│  Layer 2: 执行 + 汇报（异步知会）        │
│  例：跨渠道回复草稿、代码审查意见        │
│  条件：中风险、需要人知晓但不需要实时审批  │
├─────────────────────────────────────────┤
│  Layer 1: 决策请求（同步等待）           │
│  例：资源申请、方案选型、对外承诺        │
│  条件：高风险、不可逆、涉及外部承诺       │
└─────────────────────────────────────────┘
```

---

## 3. 人与 AI 的职责边界

### 3.1 人负责的（不可替代）

- **定义"什么是对的"** — 业务目标、优先级、价值观
- **承担后果** — 对外承诺、资源投入、风险接受
- **处理模糊性** — AI 无法判断的情况、灰色地带
- **关系维护** — 对人的沟通（客户、上级、同事）

### 3.2 AI 负责的（比人高效）

- **信息整合** — 跨渠道、跨时间、跨系统的信息汇聚
- **方案生成** — 基于"什么是对的"生成多个可行方案
- **执行跟进** — 持续执行、定期检查、异常发现
- **记录沉淀** — 自动记录决策过程、生成文档

### 3.3 灰色地带（协同完成）

- **判断优先级** — AI 排序 + 人确认
- **方案选型** — AI 生成候选 + 人拍板
- **异常处理** — AI 建议 + 人决策

## 4. 五种核心交互范式

### 4.1 异步决策流（P0）

**核心理念：** AI 不只是转发消息，而是给出带理由的推荐方案，用户一键决策。

```
┌──────────────────────────────────────────┐
│ 🔴 待决策 · 需要在今天 18:00 前          │
│                                          │
│ Lark · 王经理 询问下一阶段排期             │
│                                          │
│ AI 建议（基于当前项目进度）:               │
│ "建议回复：下周三开始，预计工期 2 周"      │
│                                          │
│ 理由：安全扫描修复预计周四完成，           │
│ 需要缓冲 1 天回归主线开发                  │
│                                          │
│ [采纳建议] [修改后发送] [我自己处理]       │
│ [延后处理]                               │
└──────────────────────────────────────────┘
```

**关键设计点：**
- AI 推荐方案带理由，不是空泛的建议
- 三个操作层级：采纳（1秒）、修改（轻量）、自己处理（完全接管）
- "延后处理"让用户不被打断，AI 稍后再提醒
- 时效性标记，区分紧急和普通决策

### 4.2 目标驱动对话（P1）

**核心理念：** 用户设定持续目标，AI 自主推进，而非逐条指令。

```
❌ 指令驱动："帮我查一下昨天注册用户数"
   → AI 执行 → 给一个数字 → 用户再问"那流失率呢？"

✅ 目标驱动："我需要准备周五向投资人汇报的用户增长数据，重点关注注册转化和留存"
   → AI 自主：拉取数据 → 分析趋势 → 发现异常 → 生成报告草稿 → 标注需要确认的数据点
   → 人只做决策：确认数据点 → 调整叙述角度
```

**关键设计点：**
- 目标设定一次，AI 持续推进
- 目标可拆解为里程碑，每个有独立状态
- AI 主动汇报进展，嵌入对话流
- 目标完成后归档，可回溯

### 4.3 执行看板（P2）

**核心理念：** 一眼看到所有数字员工的实时状态和等待决策的事项。

```
┌─────────────────────────────────┐
│ ⚠ 需要你介入 (2)                 │
│ [紧急] API 扩容决策   3 分钟     │
│ [普通] 索引优化审核   2 小时     │
│                                 │
│ 🤖 数字员工状态                   │
│ ● 安全审计员  工作中  漏洞扫描 45%│
│ ● 运维助手    等待决策 ⚠         │
│ ○ 代码开发    空闲               │
└─────────────────────────────────┘
```

**关键设计点：**
- 每个 Agent 的状态一目了然（运行中/等待决策/已完成/异常）
- 等待决策的事项高亮，不被其他信息淹没
- 点击某个 Agent → 右侧 Drawer 展示完整执行链路

### 4.4 渐进式自主（贯穿所有范式）

**核心理念：** 同一个任务中，不同环节的自主程度不同。

```
任务：处理客户反馈

[全自主] 收集各渠道客户反馈 → 分类归纳 → 生成摘要
    ↓
[AI 建议] 发现 3 个紧急问题 → 生成处理方案草稿 → 推荐优先级
    ↓
[人决策] 确认优先级和方案 → 审批资源投入
    ↓
[全自主] 按方案执行修复 → 通知客户 → 跟踪闭环
```

**视觉区分规则：**
- 全自主：折叠的灰色摘要条，不占用注意力
- AI 建议：带推荐标记的卡片，需要扫一眼
- 人决策：醒目的交互卡片，必须操作
- 执行确认：已完成打勾，可回溯查看

### 4.5 上下文空间（P3）

**核心理念：** AI 记住所有决策和知识，跨会话复用。

```
┌─────────────────────────────────┐
│ 项目：Q2 安全加固               │
│                                 │
│ 关键决策记录：                    │
│ · 04-01 蓝绿部署（你）           │
│ · 03-28 预算 28→24.5 万          │
│ · 03-25 周四前完成修复            │
│                                 │
│ AI 积累的业务知识：               │
│ · 王经理偏好周三沟通              │
│ · 部署窗口：周二/四 22:00-23:00  │
└─────────────────────────────────┘
```

**关键设计点：**
- 决策历史自动记录（何时、为什么、结果如何）
- AI 主动积累业务知识（沟通偏好、流程规则）
- 新任务自动关联相关上下文

## 5. 信息架构

### 5.1 三栏布局

```
┌──────────┬──────────────────┬─────────────────┐
│ 通知栏    │  主对话区          │ Drawer          │
│ 240px    │  flex-1           │ 按需展开         │
│          │                  │                 │
│ · 普通通知 │  · 目标状态条     │ · 执行看板(默认) │
│ · 决策请求 │  · 对话消息流     │ · 决策详情       │
│          │  · 进展汇报       │ · 目标追踪       │
│          │  · 决策卡片       │ · 上下文空间     │
│          │  [Composer]       │ · 任务详情       │
└──────────┴──────────────────┴─────────────────┘
```

### 5.2 通知栏职责

- **普通通知**：点击 → Drawer 打开 notification-detail（完整上下文 + 回复）
- **决策请求**：高亮标识（脉冲动画），点击 → Drawer 打开 decision-detail（推荐方案 + 操作按钮）

### 5.3 Drawer 职责扩展

| 类型 | 职责 | 触发方式 |
|------|------|---------|
| task-monitor | 数字员工工作台（默认） | 常驻 |
| decision-detail | 决策请求详情 | 通知栏点击决策通知 |
| notification-detail | 通知上下文详情 | 通知栏点击普通通知 |
| goal-tracker | 目标进度追踪 | 对话中点击目标卡片 |
| context-space | 项目上下文空间 | 主动打开 |
| task-detail | 任务详情 | 看板点击任务 |
| collaboration-chain | 协作链路 | 相关入口 |
| 其他 | 执行日志/代码/数据 | 相关入口 |

## 6. 技术架构

### 6.1 新增 Domain 模型

| 模型 | 文件 | 职责 |
|------|------|------|
| `DecisionRequest` | `domain/agent/DecisionRequest.ts` | AI 决策请求实体，含推荐方案、备选方案、时效性 |
| `UserGoal` | `domain/agent/UserGoal.ts` | 用户目标实体，含里程碑、进展更新 |
| `ContextSpace` | `domain/agent/ContextSpace.ts` | 上下文空间，含决策历史、知识库 |

### 6.2 扩展现有模型

| 模型 | 变更 |
|------|------|
| `Notification` | type 新增 `'decision'` |
| `AgentRuntime` | status 新增 `'awaiting-decision'`，新增 `pendingDecisionIds` |
| `MessageBlock` | 新增 `'decision-request'`、`'goal-progress'` 两种 block |
| `DrawerContent` | type 新增 `'decision-detail'`、`'goal-tracker'`、`'context-space'` |

### 6.3 分层职责

- **Domain**：新增 DecisionRequest、UserGoal、ContextSpace 不可变实体
- **Infrastructure**：Mock 数据工厂补充决策请求、目标、上下文数据
- **Application**：openclawStore 新增 decisionRequests、goals 状态；事件总线新增决策事件
- **Presentation**：新增 DecisionRequestCard、DecisionDetailContent、GoalTrackerContent、ContextSpaceContent

## 7. 实施路径

| 阶段 | 内容 | 核心交付 |
|------|------|---------|
| P0 | 异步决策流 | DecisionRequest 模型 + 对话内决策卡片 + Drawer 决策详情 |
| P1 | 目标驱动对话 | UserGoal 模型 + Composer 目标状态条 + Drawer 目标追踪 |
| P2 | 执行看板 | 重构 TaskMonitorContent → 数字员工工作台 |
| P3 | 上下文空间 | ContextSpace 模型 + Drawer 上下文视图 |

每个阶段独立可交付，不依赖后续阶段。
