# Claw Farm 产品规划与功能说明

> 版本 1.0 | 2026-04-15
> 定位：**全景导读文档** — 阅读此文档后，应能理解 Claw Farm 的产品全貌、功能体系、平台归属决策、以及各详细文档之间的关系。

---

## 1. 产品定位

### 1.1 一句话定义

**Claw Farm 是企业 AI 搭档平台的运行时引擎** — 让企业员工在 WPS 协作 IM 中拥有自己的 AI 数字搭档（ClawMate），由后台自动管理 AI 实例的创建、运行和资源回收。

### 1.2 用户视角

```
员工在 WPS 协作 IM 中发送消息
  → ClawMate 自动响应（第一次使用时会完成认领流程）
  → ClawMate 能帮员工：查日程、订会议室、发邮件、读云文档、生成图片、写代码
  → 员工不需要知道底层发生了什么
```

### 1.3 系统视角

```
WPS 协作 IM (Webhook) ──▶ Claw Farm (Go 服务)
                           ├─ 消息路由：识别用户，分发到对应 AI 实例
                           ├─ 实例管理：在 K8S 集群中创建/启动/停止 AI 实例
                           ├─ 配置注入：为每个实例写入模型配置、行为规范、工具清单
                           └─ 双向通信：通过 WebSocket 实现 IM ↔ AI 实例的实时消息
                                ▼
                          OpenClaw Pod (每用户独立)
                           ├─ AI Agent 引擎（Node.js）
                           ├─ 持久化存储 (PVC)：记忆、项目文件、Skill
                           ├─ claw-farm-channel 插件：WPS 消息收发
                           ├─ OpenViking 插件：长期记忆引擎
                           └─ 预装 Skill：办公套件(WPS)、CI/CD(eZone)、图片生成等
```

### 1.4 三个平台的角色

整个 AI 搭档平台由三个子系统协作：

| 平台 | 项目 | 核心职责 | 用户 |
|------|------|---------|------|
| **运行时引擎** | claw-farm (Go) | 消息路由 + 实例编排 + 配置注入 | 无直接用户（服务端） |
| **管理后台** | dcf-light-bot 管理后台 | 配置管理 + 运营监控 + 权限控制 | 平台管理员 |
| **租户运营平台** | dcf-light-bot 租户平台 | 租户自助配置 + 品牌定制 + 用量查看 | 租户管理员 |

### 2.3 优先级矩阵

按照**运营痛点严重程度**排序：

```
紧急度 ──────────────────────────────────────────▶
高  │ ① 模型管理(11,13,15)     ← 每次新增模型都要改代码部署
    │ ② 模板管理(17,18,19)     ← 每次调引导话术都要改代码部署
    │ ③ Skill 管理(20,21)      ← 每次增减 Skill 都要改代码部署
    │ ④ 白名单/管理员(1,2)     ← IM 命令管理低效，管理员改需重部署
中  │ ⑤ 默认模型(12)+资源规格(6) ← 环境变量管理，改需重部署
    │ ⑥ 系统消息(24)+广播(25)   ← 文案调整需改代码
    │ ⑦ 实例参数(5,8,22,23)    ← 环境变量管理，改需重部署
低  │ ⑧ 凭证管理(30,31)        ← 多租户前不紧急
    │ ⑨ 卡片模板(28,29)        ← 变动频率低
    │ ⑩ 消息记录(26)+日志(32)   ← 运维增强，非阻塞
```

## 3. 功能抽象与设计决策

### 3.1 核心抽象：从"硬编码"到"可配置"

claw-farm 的 32 个功能点可以抽象为 **6 个配置域**：

| 配置域 | 包含功能点 | 抽象方式 | 存储形式 |
|--------|----------|---------|---------|
| **模型注册中心** | #11 模型列表 + #13 提供商 + #15 路由 + #12 默认模型 | 将硬编码的模型数组改为 DB 表，支持 CRUD | `llm_models` + `llm_model_capabilities` + `llm_model_routes` 三表联合 |
| **内容模板引擎** | #17 AGENTS.md + #18 BOOTSTRAP.md + #19 TOOLS.md + #16 openclaw.json | 将 Go 代码中的 heredoc 提取为 DB 中的模板记录，支持版本管理和回滚 | `content_templates` + `content_template_versions` 两表 |
| **Skill 配置中心** | #20 预装列表 + #21 白名单 | 将硬编码数组改为 DB 表，支持勾选式管理 | `instance_skills` 单表（预装 / 白名单两个标记位） |
| **消息文案库** | #24 系统消息 + #4 拒绝文案 | 将散布在 Go 代码中的中文字符串提取为 key-value 表 | `system_messages` 单表 |
| **准入控制中心** | #1 白名单 + #2 管理员 | 将 IM 命令 / 环境变量管理改为 DB + Web 界面 | `im_whitelist` + `system_admins` 两表 |
| **运行参数中心** | #5 实例上限 + #6 资源规格 + #7 存储 + #8 空闲回收 + #22 上下文裁剪 + #23 记忆引擎 | 将环境变量管理改为 DB key-value 表 | `system_configs`（已有）+ `openclaw_configs`（已有） |

**管什么**：租户级配置、品牌定制、用量查看

| 功能模块 | 管理内容 | 与管理后台的关系 |
|---------|---------|---------------|
| 白名单 | 本租户的用户准入白名单 | 管理后台管全局，租户管自己的 |

#### 运行时引擎（claw-farm）— 执行层

**不管配置，只管执行**。改造后 claw-farm 的职责收窄为：

| 职责 | 说明 |
|------|------|
| 从管理后台拉取配置 | 启动时 + 定时轮询，缓存在内存中 |
| Webhook 接收和验证 | 签名校验、AES 解密、防重放（安全敏感，不外部化） |
| 消息路由 | 用户消息 → 对应实例（路由逻辑不外部化） |
| K8S 编排 | Pod/PVC/Service 创建/删除（编排逻辑不外部化） |
| 配置注入 | 使用管理后台的配置生成 openclaw.json 和模板文件 |
| WebSocket Hub | 实时双向通信（通信协议不外部化） |
| 空闲回收 | 根据管理后台配置的超时参数执行回收 |

## 4. 功能详细说明

以下按功能域逐一说明每个功能的**产品行为**，帮助开发者理解业务逻辑。

### 4.1 用户准入管理

#### 白名单

**用户故事**：管理员希望控制哪些员工可以使用 AI 搭档，通过邮箱前缀匹配。

**行为规则**：
- 用户在 WPS IM 中向机器人发消息时，系统提取用户的企业邮箱前缀（如 `zhangsan`）
- 检查该前缀是否在白名单中
- 不在白名单 → 回复拒绝文案，不创建实例
- 管理员用户隐式在白名单中（无需单独添加）
- 支持批量导入（CSV 文件，每行一个邮箱前缀）

**数据**：邮箱前缀（如 `zhangsan`）+ 添加人 + 添加时间

#### 管理员

**用户故事**：超级管理员希望授权某些人执行管理操作（重启所有实例、广播通知、白名单管理）。

**行为规则**：
- 管理员通过邮箱前缀识别
- 管理员可执行：`/whitelist`、`/restart-all`、`/notify`、`/new-workspace` 命令
- 当前无分级（超管和普通管理员不分），未来需支持 RBAC

#### 未注册用户拦截

**行为规则**：
- 用户通过白名单检查后，系统查询 clawhub 平台的 User + UserApiToken 表
- 如果用户未在平台注册，发送注册链接引导注册
- 注册完成后才能使用 AI 搭档

### 4.2 实例生命周期管理

#### 实例创建流程

```
用户发消息 → 检查缓存中是否有该用户的实例
  ├─ 有且 Running → 直接转发消息
  ├─ 有且 Creating/Starting → 消息入队等待
  ├─ 有且 Stopped → 重新启动（复用 PVC）
  ├─ 有且 Failed → 清理后重新创建
  └─ 没有 → 检查实例上限 → 创建新实例
              创建流程：
              1. DB 分配工号（employeeNumber）
              2. 创建 PVC（持久化存储）
              3. 创建 Pod（init 容器安装插件和写配置 → 主容器启动 OpenClaw）
              4. 创建 Service（ClusterIP）
              5. 等待 Pod Ready → WebSocket 连接建立 → 刷新排队消息
```

#### 资源规格

每个 AI 实例是一个 K8S Pod，资源规格如下（当前全局统一）：

| 资源 | 当前值 | 说明 |
|------|--------|------|
| CPU Request | 1000m (1 核) | Pod 最低保障 |
| CPU Limit | 2000m (2 核) | Pod 最高可用 |
| Memory Request | 2Gi | Pod 最低保障 |
| Memory Limit | 2Gi | Pod 最高可用 |
| Storage | 1Gi | PVC 持久化存储 |
| StorageClass | dcf-juicefs-sc | JuiceFS 分布式存储 |

**多租户后**：按套餐提供 S/M/L 规格（对应 Tenant.js 中的 free/standard/enterprise 配额）。

#### 空闲回收

**行为规则**：
- 后台每 60 秒扫描一次所有运行中的实例
- 如果实例超过 `IDLE_TIMEOUT`（默认 7200 秒=2 小时）没有用户活动 → 删除 Pod（PVC 保留）
- 回收前通过 WPS IM 通知用户："长时间未使用，实例已休眠。下次发消息时会自动唤醒。"
- 下次用户发消息时自动重新创建 Pod（PVC 中的数据还在）

#### 工作空间轮转

**行为规则**：
- 管理员执行 `/new-workspace [email]` 命令
- 旧 PVC 标记 `isActive=0`（数据永久保留）
- 创建新 PVC + 新 Pod，用户获得全新的工作空间
- **问题**：旧 PVC 永远保留，无清理策略

### 4.3 模型与 AI 配置

#### 模型体系

系统当前支持 10 个模型，分属两个 Provider：

| Provider | API 格式 | 模型 | 用途 |
|----------|---------|------|------|
| litellm-am | anthropic-messages | mcs-5, mco-4, mcs-1, mch-1 | Claude 系列对话模型 |
| litellm-oc | openai-completions | qwen-plus-latest, qwen-max-latest, qwen3-235b-a22b, qwen-vl-max-latest, mgg-9, text-embedding-v3 | Qwen/Gemini 对话 + 多模态 + 嵌入 |

所有模型通过统一的 LiteLLM 代理访问，每个用户有独立的虚拟 API Key。

#### 模型路由规则

模型 ID 到 Provider 的映射：
- `mcs*` / `mco*` / `mch*` → litellm-am（Anthropic Messages 格式）
- 其他所有 → litellm-oc（OpenAI Completions 格式）

#### 默认模型

| 用途 | 当前默认 | 说明 |
|------|---------|------|
| 对话 | mcs-5 | 通用对话场景 |
| 图片生成 | mgg-5 | 文生图场景 |
| 视觉理解 | qwen-vl-max-latest | 图片理解/分析场景 |

#### LiteLLM Key 管理

- 每个用户在实例创建时自动获得一个 LiteLLM 虚拟 Key
- Key 限制只能访问系统允许的模型列表
- Key 存储在 MySQL `FarmLitellmKey` 表中，支持跨重启复用
- 实例重启时 Key 自动续期（同步最新允许模型列表）

### 4.4 OpenClaw 实例配置

#### 配置注入流程

实例 Pod 使用两个 init 容器，在主进程启动前完成配置：

```
Init 1: init-plugin
  → 安装 claw-farm-channel 插件
  → 安装 portal-reader 插件
  → 从 ClawHub 安装预装 Skill (wps-cli, ezone-cli, clawhub-cli, image-gen)

Init 2: init-config
  → 写入 openclaw.json（模型配置、channel 配置、插件配置）
  → 写入 ov.conf（OpenViking 记忆引擎配置）
  → 写入 AGENTS.md（Agent 行为规范）
  → 写入 TOOLS.md（工具使用文档）
  → 写入 BOOTSTRAP.md（新用户引导脚本）
  → 写入 read-cloud-doc 技能脚本

Main: openclaw gateway
  → 读取配置文件启动 AI Agent 引擎
```

#### AGENTS.md — Agent 行为规范

定义 AI 搭档每次会话的行为准则：
- **Session 启动**：检测是否首次用户（IDENTITY.md 是否存在）→ 首次走 BOOTSTRAP.md 引导 / 非首次读取 SOUL.md + USER.md + 今日记忆
- **记忆系统**：4 层记忆（索引 → 项目 → 经验教训 → 日志），写入规则，日志格式
- **文件发送**：通过 message tool 的 media 字段发送本地文件
- **安全规范**：不泄露数据、不执行破坏命令、workspace 内自由操作
- **群聊规范**：不共享用户私人信息

#### BOOTSTRAP.md — 新用户引导

7 步引导流程：
1. 打招呼 + 收集用户昵称和 Agent 名字
2. 建立身份（写 IDENTITY.md + USER.md）
3. WPS 授权（引导用户访问 https://neo.ksyun.com/authorization）
4. 认领完成（展示能力清单）
5. 写 SOUL.md（Agent 性格）
6. 次日主动问候（可选 cron）
7. 标记完成（覆写 BOOTSTRAP.md）

**关键约束**：
- 绝对禁止向用户提及内部文件名
- 尽量 3 轮对话内完成
- 用户不耐烦可跳过

#### TOOLS.md — 工具使用文档

包含：
- 运行环境说明（K8s Pod，workspace 路径）
- 消息发送方法（文本、文件、指定 chat）
- 5 个预装 Skill 的用法表
- **北京小米科技园 6-11 层会议室速查表**（level_id + 容量 + 主题）
- WPS API 时间格式约定
- mcporter 使用注意事项

**特别说明**：会议室信息属于物理世界频繁变动的数据，不应硬编码在代码中。这是模板管理的典型用例 — 管理员应能随时更新会议室列表而不需要改代码。

### 4.5 消息路由与通知

#### 消息流转

```
用户在 WPS IM 发消息
  → WPS 通过 Webhook 推送到 claw-farm
  → 签名验证 + AES-256-CBC 解密
  → 提取发送者 ID、聊天 ID、消息内容
  → 白名单检查 → 斜杠命令拦截 → clawhub 注册检查
  → 如果实例在线且 WebSocket 已连接 → 直接转发
  → 如果实例不在线 → 消息入队 → 触发实例创建/启动
  → 实例 Ready 后 → 刷新排队消息
  → AI 处理完毕 → 通过 WebSocket 返回 → claw-farm 通过 WPS API 发送回复
```

#### 系统消息文案（10 条）

| 场景 | 文案 | 触发条件 |
|------|------|---------|
| 白名单拒绝 | "暂未开放，请联系管理员开通。" | 非白名单用户发消息 |
| 实例启动中 | "正在启动中，请稍候..." | 实例不在线，首条排队消息 |
| 重启成功 | "正在重启，下次发消息时自动启动..." | 用户执行 /restart |
| 无权限 | "无权限：仅管理员可执行此操作。" | 非管理员执行管理命令 |
| 日志获取失败 | "获取日志失败: {error}" | /log 命令执行出错 |
| 无实例日志 | "实例未运行，无日志可查。" | /log 命令但实例不在线 |
| 日志为空 | "(日志为空)" | 日志内容为空 |
| 空闲回收 | "长时间未使用，实例已休眠。下次发消息时会自动唤醒。" | 空闲回收前通知 |
| 未注册 | "您尚未注册平台账号..." | clawhub 账号不存在 |
| 批量重启结果 | "已重启 {count} 个实例。" | /restart-all 执行结果 |

#### 广播通知

管理员通过 `/notify <内容>` 命令向所有在线用户广播消息，支持文本和卡片格式。广播间隔 100ms/人。

## 5. 配置覆写继承机制

多租户架构下，所有可配置项遵循统一的覆写继承规则：

```
解析优先级: 租户覆写 → 系统默认 → 硬编码 fallback

示例 — 获取"白名单拒绝"文案:
1. 查 tenant_messages 表 → 有 → 用租户定制的文案
2. 查 system_messages 表 → 有 → 用系统默认文案
3. 都没有 → 用 claw-farm 硬编码的 "暂未开放，请联系管理员开通。"
```

| 配置类型 | 系统表 | 租户覆写表 | 说明 |
|---------|--------|-----------|------|
| 模型列表 | `llm_models` | `tenant_model_quotas` | 租户从全量模型中选子集 |
| 内容模板 | `content_templates` | `tenant_templates` | 租户可覆写任意模板 |
| 系统消息 | `system_messages` | `tenant_messages` | 租户可覆写任意消息 |
| Skill | `instance_skills` | `tenant_skills` | 租户可选配 Skill |
| 白名单 | `im_whitelist` | `tenant_whitelist` | 租户独立白名单 |
| 资源规格 | `system_configs` | `tenant_instance_profiles` | 租户按套餐选规格 |
| Channel 凭证 | — | `tenant_channels` | 每租户独立，无系统级 |
