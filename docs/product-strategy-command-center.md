# DCF Command Center — 产品战略白皮书

> 面向中小企业管理层的跨渠道决策指挥中心
> 人 + Agent 协同的**企业运行时（Enterprise Runtime）**

---

## 一、产品定义

### 1.1 一句话定位

**DCF Command Center 是企业管理者的"决策驾驶舱"——聚合多渠道消息、AI 智能分拣、人机协同决策、命令下达到执行闭环追踪，一站式完成。**

### 1.2 产品本质：企业运行时

DCF Command Center **不是应用，是运行时（Runtime）**。

类比：

| 概念 | 传统世界 | DCF 世界 |
|------|---------|---------|
| **操作系统** | Windows / Linux | DCF Enterprise Runtime |
| **应用程序** | Office / Chrome | 各类 AI Agent |
| **外设驱动** | 打印机驱动 / 显卡驱动 | 渠道适配器（钉钉/飞书/企微/邮箱） |
| **用户** | 操作电脑的人 | 做决策的管理者 |

操作系统不生产应用，但它为应用提供进程调度、内存管理、I/O 统一接口。DCF 不生产 Agent，但它为 Agent 提供**任务调度、消息路由、人机交互、决策闭环**的统一基础设施。

这意味着三个关键设计原则：

1. **Agent 框架不绑定**：当前以 OpenClaw 为默认 Agent 框架，但运行时层通过标准化的 `AgentRuntimeAdapter` 接口对接上层。企业可以替换为 Dify、Coze、LangChain、AutoGen 或任何自研框架——运行时只关心"任务进、结果出"的契约，不关心 Agent 内部如何思考。
2. **渠道不绑定**：Matrix、钉钉、飞书、企微、邮箱、Slack 都只是渠道适配器（ChannelAdapter）的实现。Matrix 是默认的人机协同承载通道（因为开放协议、自托管、支持 E2EE），但它与其他渠道是平级关系，不是特殊地位。任何渠道都可以作为用户触达和人机协同的入口。
3. **运行时即价值**：Agent 可以换，渠道可以换，但运行时层的**消息归一化、决策引擎、执行编排、回执追踪**是不可替代的。这是 DCF 的真正护城河——不是某个 Agent 框架做得好，而是企业的人机协同操作系统本身。

### 1.3 不是什么

| 容易混淆的东西 | 区别 |
|--------------|------|
| 又一个 IM | DCF 不承载日常聊天，只处理需要管理者注意力的信号 |
| 又一个 Agent 平台 | DCF 不构建 Agent，只调度和指挥 Agent |
| 又一个项目管理工具 | DCF 不管任务拆解，只管决策→执行→回执的闭环 |
| 某个 IM 的插件 | DCF 是跨所有 IM 的上层运行时，不依附任何单一平台 |

### 1.4 核心隐喻

传统模式：管理者是消防员——哪个群着火跑去哪个群，每个 Agent 自己跑自己的。

DCF 模式：管理者是指挥官——坐在指挥中心（运行时），所有情报通过渠道适配器自动汇聚，各类 Agent（无论来自什么框架）统一调度，需要拍板的事项排队等候，拍板后 Agent 自动执行，执行结果自动通过原渠道回传。

## 二、目标用户

### 2.1 核心用户画像

| 维度 | 描述 |
|------|------|
| **角色** | 中小企业 CEO/VP/总监/部门负责人 |
| **团队规模** | 管辖 5-200 人 |
| **日常特征** | 同时使用 2-3 个 IM（钉钉+企微、飞书+邮箱等） |
| **核心痛点** | 每天被 200+ 条消息淹没，真正需要决策的 <10 条 |
| **决策风格** | 快速判断、不愿深度使用复杂工具 |
| **预算决策** | 万元以下可自行拍板，不走采购流程 |

### 2.2 非目标用户

- 大企业 IT 部门（他们要的是深度定制，不是标准产品）
- 一线执行人员（他们在各自 IM 中工作即可）
- 已深度绑定单一 IM 且无跨渠道需求的团队

### 2.3 用户细分与场景

| 用户类型 | 典型场景 | 核心诉求 |
|---------|---------|---------|
| **CEO/创始人** | 同时管销售群、产品群、投资人群 | 不漏重要事、快速拍板、看全局 |
| **运营负责人** | 多渠道客诉监控（钉钉+企微+邮箱） | 统一收件箱、自动分级、批量处理 |
| **项目经理** | 跨团队协调（不同团队用不同 IM） | 信息归集、任务追踪、进度汇总 |
| **技术总监** | 监控报警+团队汇报+跨部门沟通 | 报警聚合、智能降噪、一键派单 |

## 三、核心价值主张

### 3.1 价值公式

```
用户价值 = 节省的注意力成本 + 决策质量提升 + 执行效率提升
```

### 3.2 三层价值递进

#### 第一层：注意力解放（消息聚合 + 智能分拣）

**问题**：管理者每天在 3 个 IM 之间来回切换 40+ 次，其中 80% 的消息不需要他本人处理。

**解法**：
- 多渠道消息统一归集到一个收件箱
- AI 智能分类：指令类 / 汇报类 / 审批类 / 闲聊类 / 报警类
- 按紧急度和相关度排序，管理者只看排在最前面的
- 低优先级消息自动摘要，不打扰

**可量化收益**：每天减少 1-2 小时消息处理时间

#### 第二层：决策提速（AI 辅助决策 + 上下文聚合）

**问题**：做一个决策需要从 3 个群、2 封邮件、1 个文档里拼凑上下文，然后才能判断。

**解法**：
- 决策卡片：自动聚合相关上下文（关联消息、历史决策、数据指标）
- AI 生成决策建议：方案对比、风险评估、影响预判
- 一键选择：采纳 / 修改 / 拒绝 / 延后
- 决策记录自动归档，形成决策审计链

**可量化收益**：平均决策时间从 2 小时缩短到 15 分钟

#### 第三层：执行闭环（命令下达 + Agent 执行 + 回执追踪）

**问题**：决策做完了，执行全靠人盯人。消息发出去不知道谁接了、做到哪了、什么时候完。

**解法**：
- 决策自动转化为可执行任务
- 任务自动派发给 Agent 或人类执行者
- Agent 能做的事（信息收集、文档生成、数据分析）自动完成
- Agent 不确定的事自动创建人工确认节点
- 执行进度实时追踪，逾期自动提醒
- 完成回执自动发回原始 IM 渠道

**可量化收益**：任务从"发出"到"确认完成"的周期缩短 60%

## 四、产品架构

### 4.1 三层解耦架构

DCF 的核心架构理念是**上下两端可插拔，中间运行时不可替代**：

```
┌─────────────────────────────────────────────────────────────────┐
│                   上层：Agent 框架（可插拔）                      │
│                                                                  │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────────┐    │
│  │ OpenClaw │  │ Dify     │  │ Coze     │  │ 自研框架      │    │
│  │ (默认)   │  │          │  │          │  │ LangChain/etc │    │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘  └──────┬───────┘    │
│       └──────────────┴──────────────┴───────────────┘            │
│                          │                                       │
│              AgentRuntimeAdapter 标准接口                         │
│              (任务进 → 结果出，框架无关)                           │
├─────────────────────────────────────────────────────────────────┤
│              中层：DCF Enterprise Runtime（核心）                 │
│                                                                  │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │                   指挥台 (Command Dashboard)              │   │
│  │  统一收件箱 · 决策队列 · 执行看板 · 洞察面板               │   │
│  ├──────────────────────────────────────────────────────────┤   │
│  │  消息分拣引擎 │ 决策引擎(DecisionHub) │ 编排引擎           │   │
│  │  NL→Intent   │ 触发·推荐·闭环        │ 派发·追踪·回执     │   │
│  ├──────────────────────────────────────────────────────────┤   │
│  │              消息总线 (Message Bus)                        │   │
│  │  归一化消息模型 · 路由规则 · 优先级计算 · 去重 · 关联       │   │
│  └──────────────────────────────────────────────────────────┘   │
├─────────────────────────────────────────────────────────────────┤
│                   下层：渠道适配器（可插拔）                      │
│                                                                  │
│  ┌───────┐ ┌───────┐ ┌───────┐ ┌───────┐ ┌───────┐ ┌───────┐  │
│  │ 钉钉   │ │ 飞书  │ │ 企微   │ │ 邮箱  │ │ Slack │ │Matrix │  │
│  │Adapter │ │Adapter│ │Adapter │ │Adapter│ │Adapter│ │Adapter│  │
│  └───────┘ └───────┘ └───────┘ └───────┘ └───────┘ └───────┘  │
│                                                                  │
│  所有渠道平级，均为用户触达 + 人机协同的承载入口                    │
│  Matrix 作为默认渠道（开放协议·自托管·E2EE），但无特殊绑定         │
└─────────────────────────────────────────────────────────────────┘
```

### 4.2 架构核心原则

#### 原则一：运行时与 Agent 框架解耦

```
                    AgentRuntimeAdapter 接口
                    ┌─────────────────────┐
                    │ submitTask(task)     │
                    │ getTaskStatus(id)    │
                    │ cancelTask(id)       │
                    │ onTaskComplete(cb)   │
                    │ listCapabilities()   │
                    └─────────────────────┘
                           ▲
            ┌──────────────┼──────────────┐
            │              │              │
    OpenClawAdapter   DifyAdapter    CustomAdapter
```

- 运行时只通过 `AgentRuntimeAdapter` 与上层交互
- 企业可以在部署时选择 Agent 框架，也可以混合使用多个框架
- Agent 框架的升级、替换、迁移不影响运行时层

**为什么不绑定特定 Agent 框架？**

AI Agent 赛道正处于爆发期，框架迭代极快。今天的最优选择半年后可能落后。运行时层的价值在于**稳定的企业流程基础设施**，不应该被某个 Agent 框架的生命周期绑架。企业选 DCF 是选一个可以"换引擎的车"，而不是"只能加某种油的车"。

#### 原则二：渠道平级，无主次之分

```
                    ChannelAdapter 接口
                    ┌──────────────────────┐
                    │ startListening(cb)   │
                    │ sendReply(params)    │
                    │ markAsRead(id)       │
                    └──────────────────────┘
                           ▲
        ┌──────┬──────┬────┼────┬──────┬──────┐
        │      │      │    │    │      │      │
      钉钉   飞书   企微  邮箱  Slack  Matrix  ...
```

- 所有渠道实现同一个 `ChannelAdapter` 接口
- Matrix 是默认渠道（原因：开放协议、自托管能力、端到端加密、无厂商锁定），但架构上与钉钉/飞书完全平级
- 企业可以只接入一个渠道，也可以同时接入多个
- 新增渠道只需实现 adapter，无需改动运行时层

**为什么 Matrix 是默认而非唯一？**

Matrix 的开放性适合作为"自有阵地"——企业可以自托管、完全控制数据。但现实中，企业员工已经在用钉钉/飞书/企微了。逼他们迁移到 Matrix 不现实。正确的姿势是：Matrix 作为运行时内部的人机协同主通道，其他 IM 作为"哨兵渠道"向 Matrix（或直接向运行时）汇报信息。用户可以在任何渠道与系统交互。

#### 原则三：运行时层是不可替代的中间件

上层 Agent 框架可以换，下层渠道可以换，但中间层提供的能力是不可替代的：

| 运行时能力 | 上层框架做不了 | 下层渠道做不了 |
|-----------|-------------|-------------|
| 跨渠道消息归一化 | Agent 只关心结构化任务 | 每个渠道只看到自己的消息 |
| 跨 Agent 任务编排 | 单个 Agent 框架不知道其他框架的 Agent | 渠道不参与任务编排 |
| 人机决策闭环 | Agent 发起决策请求，但不管理整个生命周期 | 渠道只是传输通道 |
| 执行回执路由 | Agent 输出结果，不知道该发回哪个渠道 | 渠道不知道结果该发给谁 |
| 企业级审计 | Agent 框架各自记日志 | 渠道不记录业务操作 |

### 4.3 六大核心模块

#### 模块一：消息总线（Message Bus）

运行时的"神经系统"，所有信息的归一化枢纽。

| 组件 | 职责 |
|------|------|
| ChannelAdapterRegistry | 渠道适配器注册、生命周期管理、健康检测 |
| MessageNormalizer | 将各平台异构消息转为统一消息模型（NormalizedMessage） |
| DedupEngine | 跨渠道消息去重（同一事项在多渠道被提及） |
| PriorityScorer | 基于发送人、内容、上下文计算消息优先级 |
| MessageRouter | 根据消息类型和路由规则分发到对应引擎 |

#### 模块二：智能分拣引擎

| 组件 | 职责 |
|------|------|
| IntentClassifier | 消息意图分类（指令/汇报/审批/咨询/闲聊/报警） |
| EntityExtractor | 提取关键实体（人名、日期、金额、项目名） |
| SummaryGenerator | 低优先级消息自动摘要 |
| RoutingPolicy | 根据分类结果决定消息流向（推送/静默/Agent处理） |

#### 模块三：决策引擎

| 组件 | 职责 |
|------|------|
| DecisionHub | 统一决策入口，接收多源触发 |
| ContextAggregator | 自动聚合决策所需上下文（跨渠道、跨 Agent） |
| RecommendationEngine | 生成决策建议和方案对比 |
| DecisionLifecycle | 管理决策状态流转和超时处理 |

#### 模块四：执行编排引擎

| 组件 | 职责 |
|------|------|
| AgentRuntimeAdapterRegistry | Agent 框架适配器注册，支持多框架并存 |
| CollaborationChain | 跨 Agent 协作链编排（可跨框架） |
| TaskDispatcher | 任务自动派发（根据能力匹配选择 Agent 和框架） |
| HumanInTheLoop | 人工确认节点管理 |
| ReceiptManager | 执行回执生成和路由回传原渠道 |

#### 模块五：洞察引擎

| 组件 | 职责 |
|------|------|
| InfoFlowAnalyzer | 信息流量分析（哪个渠道噪音最大） |
| DecisionMetrics | 决策效率统计（平均耗时、积压量） |
| BottleneckDetector | 执行瓶颈识别（哪个环节常卡住） |
| ReportGenerator | 自动生成日报/周报 |

#### 模块六：企业运行时基座

| 组件 | 职责 |
|------|------|
| TenantRuntime | 多租户隔离，每个企业独立的运行时实例 |
| AuditTrail | 跨渠道、跨 Agent 的统一审计日志 |
| PermissionGate | 基于角色的操作权限管控 |
| ConfigRegistry | 企业级配置管理（渠道配置、Agent 配置、路由规则） |

## 五、竞品分析与差异化

### 5.1 竞品矩阵

```
                    面向管理者（决策）
                         ↑
                         │
       DCF Command       │     飞书/钉钉
       Center            │     (单平台 + 流程)
     ┌──────────┐        │
     │ 跨平台    │        │
     │ AI 决策   │        │
     │ 人机协同   │        │
     └──────────┘        │
  ←──────────────────────┼──────────────────────→
  跨平台聚合              │          单平台深耕
                         │
       Front / Missive   │     Slack / Teams
       (跨平台 + 客服)    │     (单平台 + 协作)
                         │
                         ↓
                    面向一线（执行）
```

### 5.2 逐一对比

#### vs 钉钉 / 飞书 / 企微

| 维度 | 钉钉/飞书/企微 | DCF Command Center |
|------|---------------|-------------------|
| **本质** | 企业 IM 平台 | 跨 IM 指挥层 |
| **消息来源** | 仅本平台 | 聚合所有平台 |
| **AI 能力** | 平台内 AI 助手 | 跨平台决策 AI |
| **锁定效应** | 强绑定，迁移成本高 | 平台无关，随时插拔 |
| **目标用户** | 全员 | 管理者 |
| **竞争关系** | **不竞争，而是寄生** — DCF 以机器人/应用形态存在于各 IM 中 |

核心差异：**钉钉们是员工的办公入口，DCF 是管理者的指挥入口。不替代，不冲突，互补共生。**

#### vs Front / Missive（海外多渠道收件箱）

| 维度 | Front / Missive | DCF Command Center |
|------|-----------------|-------------------|
| **聚合对象** | 邮箱 + 社交媒体 + 短信 | 企业 IM + 邮箱 |
| **使用场景** | 客服/销售团队对外沟通 | 管理者内部决策 |
| **AI 能力** | 自动回复、分类 | 决策建议、执行编排 |
| **Agent 协同** | 无 | 核心能力 |
| **中国市场** | 无本地化，不支持钉钉/飞书/企微 | 原生支持 |

核心差异：**Front 面向对外沟通的一线团队，DCF 面向对内决策的管理者。场景不重叠。**

#### vs Dify / Coze（AI Agent 平台）

| 维度 | Dify / Coze | DCF Command Center |
|------|-------------|-------------------|
| **定位** | Agent 构建平台（框架） | Agent 调度中心（运行时） |
| **核心能力** | 创建和训练 Agent | 编排和指挥 Agent |
| **用户** | 开发者/技术人员 | 企业管理者 |
| **与 IM 关系** | 单点接入 | 多渠道聚合 |
| **决策能力** | 无 | 核心能力 |
| **关系** | **DCF 的上游供应商之一** — Dify/Coze 构建的 Agent 可通过 AgentRuntimeAdapter 接入 DCF |

核心差异：**Dify/Coze 是 Agent 的工厂（框架层），DCF 是 Agent 的司令部（运行时层）。工厂造兵，司令部调兵。不是竞争关系，而是上下游：DCF 可以同时调度来自 Dify、Coze、自研框架的 Agent。**

#### vs n8n / Zapier（自动化工作流）

| 维度 | n8n / Zapier | DCF Command Center |
|------|-------------|-------------------|
| **自动化模式** | 机器对机器 | 人对 Agent |
| **触发方式** | 事件触发 → 自动执行 | 事件触发 → 人类决策 → 执行 |
| **人类角色** | 规则设定者 | 实时决策者 |
| **适用场景** | 确定性流程 | 需要判断的不确定性流程 |

核心差异：**n8n 自动化确定性的事，DCF 处理需要人类判断的不确定性。两者的边界是"这件事机器能自己定吗？"**

#### vs Harness（企业级编排运行时）

这是最值得深入对比的对象——Harness 和 DCF 共享同一个架构范式（**中间层编排运行时**），但应用在完全不同的领域。

| 维度 | Harness | DCF Command Center |
|------|---------|-------------------|
| **定位** | 软件交付运行时 | 业务决策运行时 |
| **编排对象** | Code → Build → Deploy → Monitor | Message → Decide → Execute → Receipt |
| **输入** | Git commit、PR、制品 | IM 消息、通知、报警 |
| **输出** | 部署完成的软件 | 完成的业务决策和执行回执 |
| **用户** | 开发者、DevOps、SRE | 业务管理者、决策者 |
| **上游集成** | GitHub、GitLab、Docker | 钉钉、飞书、企微、邮箱 |
| **下游集成** | K8s、AWS、GCP、Terraform | AI Agent（Dify/Coze/OpenClaw/自研） |
| **GTM** | 企业销售（$100K+ ACV） | 嵌入式免费 + 效果付费 |
| **市场** | 全球开发者 | 中国中小企业管理层 |

核心差异：**Harness 是"代码到生产"的运行时，DCF 是"信息到决策"的运行时。** 架构模式一致，领域完全不同。Harness 证明了"中间层编排运行时"模式在软件交付领域的可行性；DCF 要验证的是同一模式在业务决策领域是否成立。

**Harness 会进入 DCF 的领域吗？**

短期不会，原因：
1. Harness 的 DNA 是开发者工具——团队、品牌、销售体系全部围绕 Dev/Ops 构建
2. 中国 IM 生态（钉钉/飞书/企微）是 Harness 零覆盖的领域
3. 业务决策编排需要组织管理、审批流、中国企业文化等领域知识
4. Harness 的 GTM 是 $100K+ ACV 企业销售，与中小企业管理者完全不匹配

**DCF 应该从 Harness 学什么？**

Harness 不是以"DevOps Runtime"的身份起家的——它是以"Continuous Delivery as a Service"起家的。**先解一个具体的痛（CD 太难了），再扩成平台。** DCF 也不应该以"企业运行时"去面向用户——"运行时"是给开发者和投资人讲的架构故事。面向管理者，要讲的是："你的消息，60 秒内分好类、排好序、可以拍板。"

### 5.3 差异化总结

DCF Command Center 的独特价值在于占据了一个**未被满足的交叉地带**——它是企业的**运行时层**，而非应用层或框架层：

| 已有产品解决的 | 未被解决的（DCF 的机会） |
|--------------|----------------------|
| 单平台内的高效沟通 | **跨平台的信息归集** |
| AI 自动执行确定性任务 | **AI 辅助人类做不确定性决策** |
| Agent 的创建和训练（框架层） | **Agent 的调度和指挥（运行时层）** |
| 一线员工的协作效率 | **管理者的决策效率** |
| 机器对机器的自动化 | **人对 Agent 的协同** |
| 绑定特定 Agent 框架的解决方案 | **框架无关的企业运行时** |

## 六、商业模式与售卖策略

### 6.0 为什么传统 SaaS 卖法行不通

在设计售卖策略之前，必须正视一个事实：

| 传统 SaaS 假设 | 在 DCF 场景下的现实 |
|---------------|-------------------|
| 企业有对应品类的采购预算 | "跨 IM 决策平台"这个品类不存在，IT 预算里没有这一行 |
| 用户会注册试用一个新平台 | 管理者不愿意多开一个平台，他已经被工具淹没了 |
| 订阅费 ¥999-7999/月能被接受 | 中小企业管理者看到月费第一反应是"我又不是买不起钉钉" |
| 14 天免费试用能转化 | 管理者没时间学一个新工具，第 3 天就忘了 |

**核心矛盾**：产品的价值在"跨平台"，但用户的行为在"单平台内"。你不能要求用户跳出来才能体验你的价值。

### 6.1 商业模式：嵌入免费 + 效果付费 + 渠道分成

**核心原则：不要求企业采购任何东西。**

```
       不花钱                  花小钱                   别人花钱
        │                       │                        │
 ┌──────▼──────┐        ┌───────▼───────┐       ┌───────▼───────┐
 │  免费嵌入    │   →    │  效果付费      │  →    │  渠道分成      │
 │  IM内机器人  │        │  按闭环收费    │       │  SI / Agent   │
 │  基础分拣免费 │        │  管理者个人付  │       │  框架合作      │
 └─────────────┘        └───────────────┘       └───────────────┘
     Phase 1                Phase 1-2               Phase 2-3
     获客                    变现                     规模化
```

#### 层一：免费嵌入（获客引擎）

DCF 在各 IM 中以**免费机器人/轻应用**形态存在。用户不"采购"DCF，而是在飞书应用市场"安装"一个决策助理。

| 渠道 | 形态 | 免费功能 |
|------|------|---------|
| 飞书 | 应用市场上架的轻应用 | 消息分拣 + 每日摘要 |
| 钉钉 | 组织内机器人 | 消息分拣 + 每日摘要 |
| 企微 | 企业应用 | 消息分拣 + 每日摘要 |

**关键设计**：用户全程在已有 IM 内操作。"决策助理"不是一个新平台，是飞书/钉钉里的一个智能机器人。零迁移成本，零采购决策。

免费层的目的不是赚钱，是让管理者体验到**"原来 80% 的消息不需要我看"**这个 aha moment。

#### 层二：效果付费（核心变现）

免费分拣吸引用户后，**按决策闭环收费**——每完成一次"分拣 → 决策 → 执行 → 回执"收费。

| 闭环类型 | 单价 | 示例 |
|---------|------|------|
| 轻量决策 | ¥5/次 | 审批确认、简单指令下达 |
| 标准决策 | ¥15/次 | 带上下文聚合 + AI 建议的决策 |
| 复杂决策 | ¥30/次 | 跨 Agent 协作链执行 + 多步追踪 |

**为什么按效果而非按月？**

- 管理者不在乎你的"运行时"有多牛，在乎"这件事帮我省了多少时间"
- 小额按次支付，可以从个人费用报销或部门零散预算出，**不走 IT 采购流程**
- 用量直接反映价值——用得多说明真的有用，用得少说明产品还没打中痛点
- 天然的 usage-based 数据，为后续融资和合作谈判提供真实指标

**月均消费预估**：

| 用户类型 | 月均闭环数 | 月均消费 |
|---------|-----------|---------|
| 轻度管理者（5-10 人团队） | 30-50 次 | ¥150-750 |
| 中度管理者（20-50 人团队） | 100-200 次 | ¥500-3000 |
| 重度管理者（50+ 人团队） | 300+ 次 | ¥1500-9000 |

#### 层三：渠道分成（规模化引擎）

DCF 不直面所有企业用户，而是通过两类合作伙伴触达更大市场：

**合作模式 A：SI / 集成商 OEM**

| 维度 | 说明 |
|------|------|
| **合作对象** | 做企业数字化转型的系统集成商（用友/金蝶生态伙伴、行业 ISV） |
| **合作方式** | SI 将 DCF 运行时嵌入其解决方案中交付给客户 |
| **价值交换** | SI 获得 Agent 编排能力（补齐能力短板）；DCF 获得客户关系和行业认知 |
| **分成** | SI 收项目服务费；DCF 按部署企业数收运行时授权费（¥500-2000/企业/月） |

**合作模式 B：Agent 框架分发分成**

| 维度 | 说明 |
|------|------|
| **合作对象** | Dify、Coze 及其他 Agent 框架厂商 |
| **合作方式** | Agent 框架的 Agent 通过 DCF 运行时到达企业用户 |
| **价值交换** | 框架方获得企业级分发渠道；DCF 获得 Agent 生态丰富度 |
| **分成** | Agent 执行产生的费用，框架方 70% / DCF 30%（或反向，视议价力而定） |

### 6.2 分阶段执行

#### Phase 1（0-6月）：免费嵌入 + 效果付费

| 动作 | 目标 |
|------|------|
| 飞书应用市场上架"决策助理" | 零采购门槛获客 |
| 消息分拣免费，决策闭环按次收费 | 验证用户是否愿意为"省时间"付费 |
| 直接找 5-10 个管理者试用 | 打磨产品，收集真实反馈 |

**核心指标**：50 个活跃用户，月均付费 > ¥500 的用户 > 10 个

**要验证的假设**：管理者是否愿意为"消息分拣 + 决策闭环"这个价值付费。如果不愿意，需要调整价值主张，而非加大销售力度。

#### Phase 2（6-12月）：SI 渠道 + 第二渠道

| 动作 | 目标 |
|------|------|
| 找 3-5 家 SI 合作 | 通过 SI 触达其客户，借助行业认知打磨垂直场景 |
| 上线钉钉/企微适配器 | 覆盖第二、第三 IM 渠道 |
| 开源运行时核心 | 建立开发者社区，降低 SI 的接入成本 |

**核心指标**：SI 合作落地 > 3 家，通过 SI 触达企业 > 30 家

#### Phase 3（12-24月）：Agent 分发 + 商业托管

| 动作 | 目标 |
|------|------|
| 与 Agent 框架方合作 | Agent 通过 DCF 到达企业，DCF 收过路费 |
| 商业托管版上线 | 为需要 SLA / 审计 / 私有化的客户提供付费版 |
| 行业垂直模板 | 电商 / SaaS / 制造业等场景的预制决策链 |

**核心指标**：MRR ¥30万+，其中渠道分成收入占比 > 40%

### 6.3 收入结构演进

```
Phase 1    ████████████░░░░░░░░  效果付费 90% │ 其他 10%
Phase 2    █████████░░░░░████░░  效果付费 50% │ SI授权 30% │ 其他 20%
Phase 3    ████░░░░████░░████░░  效果付费 25% │ SI授权 30% │ Agent分成 25% │ 托管 20%
```

健康的收入结构应该随时间从"直接向用户收费"演进为"向生态收费"。效果付费占比下降不是因为它萎缩，而是因为渠道收入增长更快。

### 6.4 售卖话术框架

#### 面向管理者（获客）

电梯演讲（30 秒）：

> 在飞书里装一个"决策助理"，免费的。它帮你把每天 200 条消息分成"需要你拍板的"和"看看就行的"。需要决策的事，它自动整理好上下文和建议，你一键拍板，后面的执行和追踪它来管。你不用装新 APP，不用学新工具，消息分拣免费用，觉得好用了再按次付费。

话术要点：**不提"运行时"、不提"平台"、不提"采购"。** 只说"免费装一个助理"。

#### 面向 SI（渠道合作）

> 你们做企业数字化项目，客户越来越多地问"AI Agent 怎么管理"。DCF 运行时是现成的 Agent 编排基座——消息接入、决策引擎、执行追踪、审计合规都有了。你把它嵌入你的解决方案里交付，我们按企业数收授权费，你该收的项目服务费一分不少。

#### 面向 Agent 框架方（生态合作）

> 你们造了很多好 Agent，但企业不知道怎么用、怎么管。DCF 运行时帮 Agent 完成"最后一公里"——接入企业 IM、对接人类决策、追踪执行结果。你的 Agent 通过我们到达企业用户，我们按调用量分成。你专心造更好的 Agent，我来解决企业落地的脏活。

### 6.5 异议处理

| 异议 | 应对 |
|------|------|
| "又多一个工具" | 不是新工具，是飞书/钉钉里的一个免费助理。你已经在用的 IM 里多了一个机器人 |
| "消息安全怎么保证" | 消息不落我们的库，只做分类索引，原始数据留在各 IM 平台。可选私有化部署 |
| "免费的东西能好用吗" | 分拣免费，闭环收费——你只在觉得值的时候付钱。第一个月不满意，用了多少退多少 |
| "AI 不靠谱" | AI 只做建议和分拣，最终拍板的永远是你。就像秘书帮你整理文件，签字的还是你 |
| "绑定某个 AI 框架怕被锁定" | 运行时不绑框架。今天用 OpenClaw，明天想换 Dify，运行时层不用动 |
| "我们只用一个 IM" | 即使单 IM，分拣 + 决策闭环 + 执行追踪的价值依然成立。跨 IM 是锦上添花 |

### 6.6 被否决的方案及理由

| 方案 | 否决理由 |
|------|---------|
| 传统 SaaS 月费订阅（¥999-7999/月） | "跨 IM 决策平台"品类不存在，IT 预算没这一行 |
| 按 seat 收费 | 管理者人数少，收不上量 |
| 纯开源先行 | 没有用户基数时开源无人问津，先有产品再谈开源 |
| 只做单一 IM 插件 | 沦为某平台附庸，"跨平台运行时"的故事讲不了 |
| 纯 to B 直销 | 中小企业 IT 采购流程长、预算紧，ROI 周期不可控 |

## 七、核心指标体系

### 7.1 北极星指标

**每周活跃决策闭环数（Weekly Active Decision Loops）**

定义：一个完整的"消息触发 → 决策 → 执行 → 回执"闭环算一次。

选择理由：
- 涵盖了产品核心价值链的每一步

- 直接关联用户付费意愿

| **获客** | 注册→接入首个IM渠道的转化率 | >60% |

| **留存** | 月活管理者留存率 | >70% |
| **收入** | 试用→付费转化率 | >15% |

## 八、风险矩阵与应对

### 8.1 外部风险

| 风险 | 概率 | 影响 | 应对策略 |
|------|------|------|---------|
| **IM 厂商收紧 API** | 高 | 产品部分失效 | 适配器层可插拔设计；优先支持开放协议（Matrix）；与各平台保持ISV合作关系 |
| **IM 厂商自己做类似功能** | 中 | 单平台内竞争 | 跨平台聚合是我们的差异，单平台做不了；保持跨平台价值不可替代 |
| **数据安全合规趋严** | 高 | 交付成本增加 | 消息不落库架构，只索引元数据；提供私有化部署选项；尽早做等保/SOC2 |
| **大厂入场** | 中 | 市场被挤压 | 大厂做跨平台聚合有政治障碍（钉钉不会帮你聚合飞书），中立第三方反而有优势 |

### 8.2 内部风险

| 风险 | 概率 | 影响 | 应对策略 |
|------|------|------|---------|
| **消息分拣准确率不足** | 中 | 用户信任崩塌 | 初期保守分类，宁可多推不漏推；持续用用户反馈微调模型 |
| **Agent 执行不可靠** | 中 | 核心价值打折 | 前期 Agent 只做信息汇总和建议，不做自动执行；逐步开放执行权限 |
| **多渠道维护成本过高** | 高 | 研发资源消耗 | 标准化 ChannelAdapter 接口；优先做 2-3 个主流渠道；社区贡献其他适配器 |
| **PMF 验证失败** | 中 | 方向需调整 | Phase 1 控制投入（<3人月），快速验证后再加注；保持代码架构灵活可 pivot |

### 8.3 特别说明：大厂"跨平台聚合"的政治不可能性

这是 DCF 最大的结构性护城河，值得展开说明：

- 钉钉不可能帮你聚合飞书消息（阿里和字节是竞争关系）
- 飞书不可能帮你聚合企微消息（字节和腾讯是竞争关系）
- 企微不可能帮你聚合钉钉消息（腾讯和阿里是竞争关系）
- 任何一家大厂做"跨平台聚合"都意味着承认竞品的存在价值

**只有中立第三方才能做跨平台聚合。** 这不是技术壁垒，是政治壁垒——最高级别的护城河。

## 九、技术护城河

### 9.0 结构性壁垒：运行时层的不可替代性

这是 DCF 最根本的护城河，独立于时间维度：

| Agent 框架能做的 | Agent 框架做不了的（DCF 运行时的价值） |
|----------------|--------------------------------------|
| 创建和训练 Agent | 跨框架调度（Dify 的 Agent 和 Coze 的 Agent 协作） |
| 单 Agent 执行任务 | 跨 Agent 协作链编排 + 人工确认节点 |
| 对接单个渠道 | 跨渠道消息归一化 + 路由 + 回执 |
| Agent 内部状态管理 | 企业级审计、权限、多租户隔离 |

**Agent 框架的竞争越激烈，DCF 越有价值**——因为企业不想被单一框架锁定，而 DCF 提供了"换引擎不换车"的能力。这是一个越来越多 Agent 框架出现就越稳固的壁垒。

### 9.1 短期壁垒（0-12月）

| 壁垒 | 说明 |
|------|------|
| **多渠道适配器积累** | 每对接一个 IM，都是非标准化工作，后来者需要重新踩坑 |
| **消息归一化模型** | 不同 IM 的消息结构千差万别，归一化模型需要大量边界case处理 |
| **分拣模型训练数据** | 真实企业消息的分类标注数据，只有先行者能积累 |

### 9.2 中期壁垒（12-24月）

| 壁垒 | 说明 |
|------|------|
| **决策模式库** | 不同行业/角色的决策模式，随使用积累越来越准 |
| **Agent 协作链模板** | 经过验证的跨 Agent 协作模板，后来者缺乏实战检验 |
| **企业知识图谱** | 每个企业的人员关系、决策习惯、信息流向形成私有知识图谱 |

### 9.3 长期壁垒（24月+）

| 壁垒 | 说明 |
|------|------|
| **网络效应** | Agent 市场：企业贡献的 Agent 越多，平台对新企业越有吸引力 |
| **数据飞轮** | 使用量 → 分拣更准 → 体验更好 → 使用量更多 |
| **切换成本** | 企业的决策历史、协作链配置、Agent 编排都沉淀在平台中 |

### Phase 1：嵌入式 MVP + PMF 验证（第 1-6 月）

**目标**：验证"管理者愿意为消息分拣和决策闭环付费"这个核心假设

| 飞书 ChannelAdapter | 消息收取 + 回复，以轻应用形态上架飞书应用市场 |

### Phase 2：SI 渠道 + 多渠道扩展（第 7-12 月）

**目标**：通过 SI 触达更多企业，覆盖第二三 IM 渠道

| 交付物 | 说明 |
|--------|------|
| 钉钉 / 企微 ChannelAdapter | 第二、三渠道适配器 |

| SI OEM 支持 | 多租户隔离 + 白标支持 + SI 接入文档 |
| 跨渠道消息去重和关联 | 同一事项在多渠道的识别 |

**验收标准**：SI 合作落地 >3 家，通过 SI 触达企业 >30 家

### Phase 3：Agent 生态 + 规模化变现（第 13-24 月）

**验收标准**：MRR ¥30 万+，渠道分成收入占比 >40%

## 附录 A：术语表

| 术语 | 定义 |
|------|------|
| **企业运行时** | DCF 的核心层，提供消息路由、决策引擎、执行编排、审计等基础设施，上层 Agent 框架和下层渠道均可插拔 |
| **决策闭环** | 一次完整的"消息触发→管理者决策→Agent执行→回执确认"流程 |
| **指挥台** | 管理者的主操作界面，聚合所有渠道的待处理事项 |
| **消息分拣** | AI 对收到的消息进行意图分类、优先级评估和路由分发 |
| **协作链** | 多个 Agent 之间的任务委托和数据传递链路，可跨 Agent 框架 |
| **回执** | 任务执行完成后自动发回原始渠道的确认消息 |
| **渠道适配器** | 对接特定通信平台的标准化组件，所有渠道平级 |
| **AgentRuntimeAdapter** | 对接特定 Agent 框架的标准化接口，使运行时与框架解耦 |
| **人在环中** | Agent 执行链路中需要人类确认才能继续的节点 |

## 附录 C：三层解耦决策记录

### 为什么 Agent 框架可插拔？

| 考量 | 说明 |
|------|------|
| **市场现实** | AI Agent 赛道爆发期，框架迭代极快，今天最优选择半年后可能落后 |
| **客户需求** | 不同企业有不同偏好——有的用 Dify，有的自研，有的政策要求用国产框架 |
| **商业策略** | 不与 Agent 框架竞争，而是做它们的下游消费者，扩大可合作生态 |
| **技术实现** | 通过 `AgentRuntimeAdapter` 标准接口隔离，运行时只关心任务契约 |

### 为什么渠道平级、Matrix 为默认？

| 考量 | 说明 |
|------|------|
| **用户现实** | 企业员工已在各IM中，逼迁移不现实 |
| **Matrix 优势** | 开放协议、自托管、E2EE、无厂商锁定——适合作为"自有阵地" |
| **架构原则** | 渠道是运行时的"外设"，不是运行时的"内核"，任何渠道都不应有特殊地位 |
| **扩展性** | 新增渠道只需实现 `ChannelAdapter`，零侵入运行时层 |

### 为什么运行时层是核心价值？

| 考量 | 说明 |
|------|------|
| **不可替代性** | 上层框架做不了跨框架调度，下层渠道做不了跨渠道路由，只有中间层能做 |
| **稳定性** | 企业流程基础设施需要稳定，不能随 Agent 框架的迭代而动荡 |
| **切换成本** | 企业的决策历史、编排配置、审计数据沉淀在运行时层，形成粘性 |
| **商业逻辑** | 运行时按价值收费（决策闭环数），不受上下游成本波动影响 |
